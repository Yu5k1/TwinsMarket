# 仓位日志与下单表单价格跳动 — 问题分析

## 一、仓位日志不显示

### 现状

点击「仓位日志」Tab 后始终显示 "仓位日志将在平仓时记录"，即使已有仓位被平仓。

### 根因：3 处断裂

| 层级 | 文件 | 问题 |
|------|------|------|
| **后端 — 记录** | `main.py` / `api.py` | `TradeJournal.record()` **从未被调用**。平仓逻辑在 `api.py:close_position()` 中执行，但该方法没有导入 `trade_journal`，也没有在平仓完成后调用 `record()`。 |
| **后端 — 传输** | `ws_push.py` | `push_tick()` 构建的 tick_data 中**不包含 journal 字段**。前端收不到任何日志数据。 |
| **前端 — 渲染** | `frontend/index.html` | `renderJournal()` 函数是**硬编码占位符**，直接写死 "仓位日志将在平仓时记录"，不从 `S.tick` 读取任何数据。 |

### 数据流断裂示意

```
平仓成功 → api.py close_position() → 删除仓位 → 返回 {success:true}
                                              ❌ 没有调用 trade_journal.record()

WS 推送 → ws_push.py push_tick() → tick_data 中 ❌ 没有 journal 字段

前端 → renderJournal() → ❌ 硬编码占位文字，不读数据
```

### 修复思路

1. `api.py` 中，平仓成功后调用 `trade_journal.record(pos, close_price, pnl, 'manual')`
2. `ws_push.py` 中，`push_tick()` 的 tick_data 加入 `'journal': self.trade_journal.entries[:20]`
3. 前端 `renderJournal()` 根据 `S.tick.journal` 渲染真实的平仓记录列表
4. `main.py` 中将 `trade_journal` 注入到 `api.py` 和 `ws_push.py`

## 二、下单表单的强平价/损益两平价/保证金/手续费 一直在跳动

### 现状

下单区域（`#orderForm`）中的四个计算值随市场价格不断变化，数值每秒跳动多次。

### 计算方式

所有值都在前端 `recalc()` 函数（`index.html:1426`）中计算：

```javascript
const price = t.price || t.mid_price || 0;  // ← 这里
const qty = parseFloat(document.getElementById('qtyIn').value) || 0;
const lev = S.leverage;
const notional = qty * price;
const margin = notional / lev;
const fee = notional * 0.0005;

// 强平价（多头）
const liqPx = price * (1 - 1/lev + 0.01);
// 损益两平价（多头）
const bePx = price * (1 + 0.001);
```

`t.price` 来源：`ws_push.py:138` → `engine.last_price`，即**最后一笔成交价**。

### 跳动根因

| 因素 | 说明 |
|------|------|
| **价格源不稳定** | `engine.last_price` 是逐笔成交价，每次成交都变。引擎每秒约 10 笔成交，价格在 bid/ask 之间来回跳动。 |
| **每次 tick 都重算** | `recalc()` 在每个 WS tick（约 100ms）中都被 `renderAll()` 调用，频率高。 |
| **限价单也未用限价** | 即使选择了「限价」单，`recalc()` 仍然用 `t.price` 而非用户输入的限价 `priceIn.value`。 |

### 为什么这些值理论上不应该跳

| 字段 | 实际含义 | 正确做法 |
|------|---------|---------|
| **强平价** | 开仓后，基于**实际成交价**算出的清算价 | 用 `entry_price`（开仓时确定的成交价），而非变动的市场价 |
| **损益两平价** | 覆盖手续费所需的价差 | 用 `entry_price`，固定不变的基准 |
| **保证金占用** | `数量 × 成交价 ÷ 杠杆` | 用 `entry_price` 或至少用 mid_price |
| **手续费** | `数量 × 成交价 × 0.05%` | 同上 |

### 注意：仓位卡片的对应值不会跳

仓位卡片（`#posBody`）中的强平价用的是 `p.entry_price`（开仓时的固定成交价），所以不会跳动。跳动只存在于**下单表单的预估区域**。

### 修复思路

1. 市价单：用 `t.mid_price` 或 `t.mark_price` 替代 `t.price`（中间价比成交价稳定得多）
2. 限价单：用用户输入的 `priceIn.value`，而不是市场价
3. 可选：对 price 做小幅去抖（100ms 内不重复渲染）
